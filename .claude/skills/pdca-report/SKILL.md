# pdca-report スキル

プログラミングスクール向け日次PDCA報告を対話的に作成・更新する。MCP ツール（`mcp__pdca-mcp__*`）を優先的に使用し、フォールバックとして `bin/pdca` CLI を使う。

## 前提
- 作業ディレクトリは `pdca-cli` リポジトリ内であること（`bin/pdca` が動く場所）。違う場合はユーザーに `cd /Users/devk/dev/pdca-cli` を促す。
- すべての CLI 呼び出しには必ず `--json` を付けて機械可読出力を得る。
- 1日1報告の制約あり（同じ日付で `report create` を再実行するとエラー。更新は `report update`）。

## 全体フロー

以下を上から順に実行する。分岐は状況に応じて選ぶ。

### 1. 認証チェック
MCP ツール `mcp__pdca-mcp__whoami` を呼ぶ。
- 成功（user情報あり）→次へ
- エラー → MCP のログインを試みる（`mcp__pdca-mcp__login`）。それでもダメなら CLI の `bin/pdca login` を案内して停止

### 1.5. キュー復旧チェック（サーバー停止フォールバック）

`~/.pdca-mcp/queue/` に **過去の障害時に保存されたローカルキュー**があれば、サーバー復旧した今のうちに自動で本番投稿する。

**手順**:

1. キュー検出
   ```bash
   ls ~/.pdca-mcp/queue/*.json 2>/dev/null
   ```
   ディレクトリ・ファイル不在は no-op で次へ。

2. 各 JSON ファイルを Read して内容を取得し、以下を判定:
   - `mcp__pdca-mcp__report_show(date: <report_date>)` で本番側の存在確認
   - **本番不在** → `payload` を使って `mcp__pdca-mcp__report_create` を実行
     - 成功 → ファイル削除（`rm <path>`）
     - 失敗（5xx/network 継続）→ ファイルは残し、ユーザーに「サーバーまだ不安定です」と通知
   - **本番あり（同日報告済）** → `mkdir -p ~/.pdca-mcp/queue/conflicts/` し当該ファイルを移動
     - 「ローカルキュー(<date>)と本番側に既存報告があります。conflicts/ に退避しました。手動マージしてください」と通知

3. 同期件数を一行サマリで報告:
   > 復旧キューを 3 件同期しました（うち 1 件は競合のため conflicts/ に退避）

**前提**:
- キューファイル名形式: `{user_id}__report_create__{YYYY-MM-DD}.json` または `{user_id}__report_update__{YYYY-MM-DD}.json`
- 認証エラーが連発したら即座に処理を打ち切り、ユーザーに知らせる
- バリデーションエラー (4xx) のキューは flush せず conflicts/ に移動して通知（payload に問題があるため自動修復不可）

### 2. 今日の報告・週次目標・学習計画・学習時間の状況を把握
並列で取得する（MCP ツール優先）:
- `mcp__pdca-mcp__report_today`
- `mcp__pdca-mcp__goal_current`
- `mcp__pdca-mcp__plan_show`
- `mcp__pdca-mcp__study_show` (date: 今日の日付)
- `mcp__pdca-mcp__daily_show` (date: 今日の日付)

### 3. 週次目標が未設定なら設定を提案
`goal current --json` の `weekly_goal` が `null` のとき:
- ユーザーに「今週の目標が未設定です。設定しますか？」と確認
- 設定する場合、`plan show --json` のカテゴリ一覧を番号付きで提示し、**どのカテゴリに取り組むか**を選ばせる（複数選択可）
- 選択された ID を使って:
  ```bash
  bin/pdca goal create --category_ids <ID> <ID> --json
  ```
- スキップされた場合は週次目標なしのまま進む

### 4. 既存の今日の報告をチェック
`report today --json` の `report` が `null` でないとき:
- 「今日の報告は既にあります。**更新**しますか？」と確認
- 更新する場合は **第5節の質問は変更したい項目だけ**行い、`report update --date <今日> --<field> <値> --json` で更新
- 新規作成はスキップ

`report` が `null` なら新規作成モードに進む。

### 5. PDCA 4項目を1つずつ質問

ユーザーは各項目に何を書くべきかを覚えていない前提で、**項目の意味・書き方のヒント・例**を毎回提示してから聞く。AskUserQuestion は選択肢必須で使えないので通常メッセージで質問する。

**1項目ずつ**聞く。ユーザーが答えたら次の項目に進む。空回答なら聞き直す。前の回答は絶対に残す（蓄積していく）。

**重要: Plan の扱い（前日計画フロー）**

Plan は基本的に**前日のうちに翌日分を立てておく**運用とする。したがって:

- 今日の報告作成時に、既に今日の Plan が前日に登録されている可能性がある
- 手順 4 で取得した `report today --json` の `report.plan` が埋まっていれば、それを提示して**確認する**:
  ```
  前日に立てた今日のPlan:
  「<既存のplan内容>」

  この内容で進めます。変更しますか？ (変更不要なら Enter / 変更内容を書けば上書き)
  ```
- 既存 Plan が無い場合のみ、下記テンプレートで**新規に聞く**
- この運用のため、**報告送信の最後**（第8節の後）に「**明日のPlanを今のうちに立てましょう**」としてユーザーに明日の計画を聞き、未来日付の `report create` で明日分の Plan だけを先行登録する（第9節）

#### 質問テンプレート

各項目の質問は下記のフォーマットで出す（Markdown装飾して読みやすく）:

---

**① Plan（計画）** 〔1/4〕

> 今日やろうとしていたこと・学習予定です。
> 例: 「Railsチュートリアル第8章（セッション管理）を読み進める」「RSpecの基本構文を学ぶ」

**何を予定していましたか？**

---

**② Do（実行）** 〔2/4〕

> 実際にやったこと・進めた内容です。予定と違っても構いません。
> 例: 「第8章のログイン機能を実装して動作確認まで完了」「途中で詰まって第8章前半だけ」

**実際に何をしましたか？**

---

**③ Check（振り返り）** 〔3/4〕

> 気づき・学び・詰まった点・理解度の評価。うまくいった/いかなかった理由。
> 例: 「has_secure_password の仕組みを理解できた」「セッションとクッキーの違いが曖昧」

**今日の振り返りは？**

---

**④ Action（次のアクション）** 〔4/4〕

> 明日以降のアクション・次にやること・詰まりの解消方針。
> 例: 「明日は第9章のRemember me機能に進む」「クッキーについて公式リファレンスを読む」

**次は何をしますか？**

---

各回答はそのまま変数 plan / do / check / action に保持する。

### 5b. 基礎反復の内容を質問

PDCA 4項目の後に、基礎反復（ドリル・反復練習）の内容を聞く。

---

**⑤ 基礎反復** 〔追加項目〕

> 今日取り組んだ基礎反復（ドリル・反復練習）の内容です。
> 結果をそのままコピー＆ペーストしてください。
> （今日は基礎反復をしていなければ「なし」と答えてください）

**基礎反復の内容を貼り付けてください:**

---

ユーザーが貼り付けた内容はそのまま変数 `kiso_hanpuku` に保持する。「なし」「スキップ」の場合は空とする。
**基礎反復の内容は文章整理ステップを行わない**（コピー＆ペーストされた原文をそのまま保持する）。

### 5c. 学習時間を質問

基礎反復の後に、今日の学習時間を聞く。

---

**⑥ 学習時間** 〔追加項目〕

> 今日の学習時間を教えてください。**予定**と**実際**の両方を、時間スロット形式で入力してください。
> 形式: `HH:MM-HH:MM`（複数ある場合はカンマ区切り or 改行）
> 例: 「予定: 09:00-12:00, 14:00-17:00」「実際: 09:30-12:00, 14:00-16:30」

**予定の学習時間は？**（例: 09:00-12:00, 14:00-17:00）

---

ユーザーが予定を答えたら、次に実際の学習時間を聞く:

---

**実際の学習時間は？**（例: 09:30-12:00, 14:00-16:30）

---

各スロットを配列にパースして変数 `study_planned` / `study_actual` に保持する。
- `study_show` で既に今日の学習時間が登録済みなら、その内容を提示して変更有無を確認する
- 合計時間（時間・分）を自動計算してユーザーに提示する

#### 文章整理ステップ（各項目共通）

ユーザーが答えるたびに、その入力を**そのまま**保持せず、以下を行う:

1. **文脈を踏まえて表現を整理する**
   - 今週の目標・前後のPDCA項目・Planとの対応関係を踏まえて、**意味が一意に伝わる日本語**に整える
   - 例: Plan「Rails入門の20%まで終わらせる」に対して、Doで「100%までやっています」と言われたら、"100%" が Plan(20%) に対する達成率を指していることが伝わるように「Rails入門を20%地点まで完了し、今日の計画(20%)を100%達成した」のように書き換える
2. **意味の追加・誇張・省略はしない**
   - 元発言に無い情報を補足しない（例えば「良い感じ」と勝手に評価を加えない）
   - 元発言にあった情報を削らない
3. **整理結果をユーザーに提示して確認を取る**:
   ```
   以下のように整理しました:
   「<整理後テキスト>」

   これでOKですか？ (OKなら「はい」/ 修正があれば書き換えてください)
   ```
4. ユーザーがOKしたら確定。修正指示があれば反映して再確認。

5. **曖昧で一意に解釈できない場合**は整理せず、まずユーザーに意味を確認する質問を投げる（例:「"100%" は Plan(20%) の達成率ですか、それとも Rails入門全体の100%という意味ですか？」）

#### Action 項目の % 自動提案

Action（次にやること）を聞く際、**翌日のターゲット%を計算して提案**する。目的は「目標(100%)までを残り日数で等間隔に割り付ける」こと。

**計算ロジック**:

1. 週次目標から `week_end_date` を取得（`goal current --json` の `week_end_date`）
2. 今日の日付（`date +%Y-%m-%d`）と `week_end_date` から **残り日数 R**（翌日〜週末まで、inclusive）を算出
   - 例: today=木曜、week_end=日曜 → R = 3（金・土・日）
3. 今日時点の累計進捗 **P**（Plan/Do で報告された累計%）を把握
4. **翌日ターゲット = P + (100 − P) / R**（小数第1位で四捨五入。%整数でもOK）

**提案フォーマット**:

```
今日時点で Rails入門 <P>% 完了。
残り <R> 日（<week_end_date> まで）で 100% を目指すと、等間隔配分では:
  - 明日: <翌日ターゲット>%
  - 明後日: <+増分>%
  - ...
  - <週末>: 100%

Action の案: 「Rails入門を <翌日ターゲット>% 地点まで進める」

この案でOKですか？ 別の%や、別の行動を書きたければ書き換えてください。
```

ユーザーが数値を変えたり別の表現にしたい場合はそれを採用。OKならそのまま Action に確定。

**注意**:
- 進捗%の取得: シンプルには報告内容から読み取る。`plan show --json` のカテゴリ `completed` はブール値なので中途進捗は拾えないため、**今日の Plan/Do 内で言及された数値をそのまま使う**
- 残り日数が0（今日が week_end_date）なら「今週の目標は今日が締切です」と伝え、%提案はスキップ
- 週次目標が無い場合は%提案せず、自由記述で聞く

### 6. 学習状況 (status) の判定
- デフォルトは `green`
- ユーザーが明示的に「詰まっている」「止まっている」などと言った場合、または Check/Action から明らかに困難が読み取れる場合は確認する:
  - やや詰まり: `yellow`
  - 完全停止: `red`
- ユーザーが指示した場合のみ変更する

### 7. サマリー表示と確認

以下の形式で全内容を表示:
```
--- 報告内容の確認 ---
日付       : YYYY-MM-DD (今日)
status     : green
Plan       : ...
Do         : ...
Check      : ...
Action     : ...
基礎反復   : ...（または「なし」）
学習時間予定: HH:MM-HH:MM, ... (合計 X時間Y分)
学習時間実際: HH:MM-HH:MM, ... (合計 X時間Y分)
-------------------
この内容で送信しますか？ (yes/no)
```

ユーザーが修正を求めたら、該当項目を聞き直してから再サマリー。

### 8. 送信

新規作成:
```bash
bin/pdca report create \
  --status <status> \
  --plan <plan> \
  --do <do> \
  --check <check> \
  --action <action> \
  --json
```

更新の場合:
```bash
bin/pdca report update --date <YYYY-MM-DD> \
  --<field> <値> \
  --json
```

成功したら JSON を軽くパースしてユーザーに結果を伝える（例: 「報告を送信しました。report_id: 123」）。エラーなら原因を伝えて再試行を提案。

#### サーバー停止時のフォールバック（ローカルキュー保存）

`report_create` / `report_update` が以下のエラーを返した場合、**諦めずにローカルキューに保存**する:

- HTTP 5xx (`SERVER_ERROR`)
- ネットワーク到達不能 (`NETWORK_ERROR`)
- タイムアウト

**手順**:

1. ディレクトリ作成:
   ```bash
   mkdir -p ~/.pdca-mcp/queue
   ```

2. ファイル `~/.pdca-mcp/queue/{user_id}__{operation}__{report_date}.json` に以下を保存:

   ```json
   {
     "operation": "report_create",
     "report_date": "YYYY-MM-DD",
     "payload": {
       "learning_plan": "...",
       "learning_do": "...",
       "learning_check": "...",
       "learning_action": "...",
       "learning_status": "green",
       "curriculum_name": "...",
       "code_content": "..."
     },
     "queued_at": "ISO8601",
     "user_id": <id>,
     "user_email": "...",
     "attempts": 0,
     "last_error_code": "SERVER_ERROR",
     "last_error_at": "ISO8601"
   }
   ```

3. `study_log` / `goal_update` も同様に失敗したら `queue/` 内に同形式で別ファイル保存（operation を `study_log` / `goal_update` に変える）

4. ユーザーに通知:
   ```
   ⚠️ サーバーが応答しないため、ローカルに保存しました（~/.pdca-mcp/queue/）。
      復旧後、次回 /pdca-report 実行時に自動送信されます（Step 1.5 で処理）。
   ```

5. **Step 9（docs/reports/ 保存）は queue 状態でも必ず実行**（人間用バックアップとして残すため）

**重要**:
- `bin/pdca` CLI フォールバックも失敗した場合のみ queue 書き込みを行う（CLI が成功すれば queue 不要）
- queue にあるファイルは Step 1.5 で次回起動時に自動 flush 試行される
- バリデーションエラー (4xx) は queue しない（仕様起因のため再試行不可、ユーザーに修正を促す）

#### 週次目標の進捗率を自動更新

PDCA 報告の送信と同時に、Do の内容から進捗%を読み取り、週次目標の進捗率を自動更新する。

**ロジック**:
1. Do の確定テキストから数値（%）を抽出する（例: 「30%地点まで進めた」→ 30）
2. `goal_current` で取得済みの週次目標ID・アイテムIDを使い、`mcp__pdca-mcp__goal_update` で進捗率を更新する:
   ```
   mcp__pdca-mcp__goal_update(id: <weekly_goal_id>, items: [{"id": <item_id>, "progress": <抽出した%>}])
   ```
3. 複数カテゴリが週次目標にある場合、Do の内容からどのカテゴリの進捗か判定して対応するアイテムを更新する
4. Do から%が読み取れない場合はスキップ（ユーザーに確認しない）

#### 学習時間の記録

PDCA 報告の送信と同時に、学習時間を MCP ツールで記録する:

```
mcp__pdca-mcp__study_log(date: "YYYY-MM-DD", slot_type: "planned", slots: ["09:00-12:00", "14:00-17:00"])
mcp__pdca-mcp__study_log(date: "YYYY-MM-DD", slot_type: "actual",  slots: ["09:30-12:00", "14:00-16:30"])
```

学習時間が「なし」「スキップ」の場合は記録しない。

### 9. ローカルにドキュメント保存

送信成功後、同じ内容を **`docs/reports/YYYY-MM-DD.md`** として保存する（pdca-cli リポジトリ直下の `docs/reports/` ディレクトリ。無ければ作成）。

**目的**: API と別にローカルで振り返り・検索できる資産を残す。後から `docs/reports/` を grep して過去ログを辿れる。

**フォーマット**:

```markdown
# YYYY-MM-DD (曜日) PDCA報告

- report_id: <id>
- status: <emoji> <status>
- 週次目標: <週次目標の内容> (<week_start_date> 〜 <week_end_date>)

## Plan
<plan本文>

## Do
<do本文>

## Check
<check本文>

## Action
<action本文>

## 基礎反復
<基礎反復の内容>（なしの場合は「なし」）

## 学習時間
- 予定: HH:MM-HH:MM, ... (合計 X時間Y分)
- 実際: HH:MM-HH:MM, ... (合計 X時間Y分)

## 明日のPlan（先行登録）
<明日の plan本文> (report_id: <id>)
```

status のemoji: green=🟢, yellow=🟡, red=🔴

**書き込みルール**:
- 同日のファイルが既に存在する場合は**上書き**（update フローでも最新内容を反映するため）
- 「明日のPlan」は第10節で先行登録した場合のみ追記。スキップなら省略
- ファイル名は必ず YYYY-MM-DD 形式の ISO 日付

### 10. 明日のPlanを先行登録（前日計画フロー）

今日の報告送信に成功したら、続けてユーザーに聞く:

> **明日（YYYY-MM-DD）のPlanを今のうちに立てましょう。**
> 今週の目標「<週次目標の内容>」を踏まえ、明日何をやりますか？
> 例: 「Rails入門の30%地点まで進める」「<前回の続き>」
> （スキップしたければ「スキップ」と答えてください）

ユーザーが入力したら、明日の日付で **Plan のみ** を先行登録する:

```bash
bin/pdca report create --date <YYYY-MM-DD 明日> --plan <plan内容> --status green --json
```

※ 未来日の report は Plan のみ保存される（Do/Check/Action は無視される仕様）。status は未来日では null になるが、フラグは付けて問題ない。
※ フラグ名は `bin/pdca report help create` で最新の仕様を確認してから使う（現状 `--date`）。

スキップされた場合は何もせず終了。

### 11. キュー状況確認（オプション）

ユーザーが「キュー」「pending」「未送信」などと聞いたら以下を実行:

```bash
ls ~/.pdca-mcp/queue/*.json 2>/dev/null
```

各ファイルを Read して以下を一覧表示:
- 報告日付（`report_date`）
- 保存日時（`queued_at`）
- 試行回数（`attempts`）
- payload の `learning_plan` 冒頭一行

競合ファイル（手動対応待ち）は別枠で表示:

```bash
ls ~/.pdca-mcp/queue/conflicts/*.json 2>/dev/null
```

**緊急破棄が必要な場合**:
```bash
rm ~/.pdca-mcp/queue/<file>.json
```
ただし破棄前に必ず `docs/reports/{date}.md` の存在確認を行い、人間用バックアップが残っていることを確認する。

## 注意事項

- **引数エスケープ**: Plan/Do/Check/Action にスペースや改行が含まれるため、シェル実行時は必ずクォートする。改行を含む場合は `$'...\n...'` またはシェル変数経由で渡す。
- **未来日**: 報告日が未来なら Plan のみ保存され、Do/Check/Action は無視される。今日以外の日付を指定するケースは通常ないので、明示指定されない限り今日の日付で固定。
- **--force によるgoal上書き**: 既存の週次目標がある状態で再作成する場合は `--force` が要るが、既存目標が消えるため**必ずユーザーに上書きの確認を取ってから実行**する。
- **終了コード**: 0=成功、1=認証/ネットワーク、2=バリデーション。非0なら停止してユーザーに報告。
- **プライバシー**: 学習内容は個人の振り返り。勝手に内容を要約・変換・誇張しない。ユーザーが書いた文字列をそのまま送る。

## 使用例

ユーザー: `/pdca-report`

Claude: 状況を確認します → 今日の報告なし・週次目標なし → 「今週の目標を先に設定しますか？」→ 「カテゴリから選んでください: 1) Ruby基礎 ...」→ 選択 → `goal create` 実行 → 「ではPlanから聞きます。今日やろうとしたことは？」→ Plan入力 → Do入力 → ... → サマリー → 確認 → `report create` 実行 → 完了報告
